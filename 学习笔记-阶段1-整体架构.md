# Research Copilot 学习笔记 — 阶段 1：整体架构

## 一、这个项目到底是什么？

一句话：**用户输入一个研究方向（如"class incremental learning"），系统自动检索论文、
解析PDF、生成文献综述、发现研究空白、拆解方法重组新方案、预测趋势，全部自动化。**

用户能看到的：Web UI 上输入 query → 看到综述、Gap 分析、方法重组、趋势评分。

## 二、为什么用多 Agent 而不是一个大 prompt？

| 维度 | 单 Agent（一个大 prompt） | 多 Agent |
|------|--------------------------|----------|
| 幻觉 | prompt 太长太杂，LLM 容易顾此失彼 | 每个 Agent 职责单一，prompt 聚焦，幻觉更少 |
| 调试 | 输出不对，不知道是哪个环节的问题 | 每个 Agent 独立输入/输出，定位问题快 |
| 质量 | 自己给自己打分没有意义 | Critic 单独评估，Self-Reflection 真正有效 |
| 扩展 | 加功能要改整个 prompt | 加一个 Agent 节点即可 |

**面试一句话**：单一职责 → prompt 聚焦 → 幻觉少 → 可调试 → 可扩展。

## 三、数据怎么流转？（最重要，面试必问）

```
用户输入 "class incremental learning"
    │
    ▼
[1] Intent ─── 识别意图：search/qa/summarize/refine
    │           ↓ search/summarize/refine → coordinator
    │           ↓ qa → analysis（跳过检索，直接用已有论文库回答）
    ▼
[2] Coordinator ─── 规划：把 "class incremental learning" 改写成多个检索 query
    │                 + 列出重点关注方向
    │                 例如：["class incremental learning", "CIL continual learning",
    │                       "exemplar-free incremental learning"]
    ▼
[3] Search ─── 对每个改写后的 query，同时查 arXiv + Semantic Scholar
    │           → 去重（按标题 + paper_id）
    │           → 筛选（引用数≥5，年份≥2022，最多保留10篇）
    ▼
[4] Analysis ─── 对筛选后的论文：下载PDF → 解析 → 切chunk → FAISS向量化
    │             同时用 query 做向量检索，拿到 top-20 相关 chunk 作为 RAG 上下文
    ▼
[5] Writing ─── 两种模式：
    │           - 综述模式：papers_context + rag_context → 生成文献综述
    │           - QA模式：rag_context → 回答用户问题
    │           - 修订模式（Critic 不通过时）：原综述 + feedback → 修订
    ▼
[6] Critic ─── 从三个维度打分：
    │           - 覆盖度（有没有遗漏重要方向）
    │           - 准确性（有没有编造不存在的方法/结论）
    │           - 连贯性（结构是否合理、逻辑是否通顺）
    │           总分≥阈值 → 通过 → 进入 Novelty
    │           总分<阈值 → 反馈给 Coordinator 重新规划（最多2次）
    ▼
[7] Novelty ─── 四维 Gap 分析：方法论空白/跨域机会/数据集局限/评估缺陷
    │             跨域迁移：从其他领域搬方法过来
    │             新颖性验证：对生成的思路做查重
    ▼
[8] Decomposition ─── 把论文方法拆成原子组件（编码器/损失/训练策略...）
    │                   跨论文重组新方案 + 验证兼容性
    ▼
[9] Trend ─── 不用本地论文！独立做大规模检索：
    │           - Semantic Scholar 按年份分桶，拿 total（领域级总量）+ 50篇样本
    │           - arXiv 按年份分桶，50篇样本
    │           → 时间线分析 → 方法演化追踪 → 四维趋势评分
    ▼
[10] Profile ─── 基于历史查询 + 论文库构建知识图谱
    │              核心领域掌握度 + 方法深度 + 盲区 + 推荐方向 + 研究风格
    ▼
[11] Memory ─── 持久化：论文库存盘 + 用户偏好记录
    │
    ▼
  结束，结果返回给前端
```

## 四、LangGraph 怎么编排的？

核心文件：`agent/graph.py`

### 关键概念

1. **StateGraph**：LangGraph 的图结构，每个节点是一个函数，边定义执行顺序
2. **ResearchState**：全局状态字典（TypedDict），所有节点共享读写
3. **条件路由**：`add_conditional_edges` — 根据状态决定走哪条边

### 代码结构

```python
# 1. 定义节点函数
def intent_node(state): ...      # 每个函数接收 state，返回更新的字段
def coordinator_node(state): ...
...

# 2. 创建图
graph = StateGraph(ResearchState)

# 3. 添加节点
graph.add_node("intent", intent_node)
graph.add_node("coordinator", coordinator_node)
...

# 4. 定义边（执行顺序）
graph.set_entry_point("intent")
graph.add_edge("coordinator", "search")   # 固定边：A → B
graph.add_conditional_edges(              # 条件边：根据状态选择
    "intent", route_by_intent,
    {"coordinator": "coordinator", "analysis": "analysis"}
)
graph.add_conditional_edges(
    "critic", route_by_critic,
    {"novelty": "novelty", "coordinator": "coordinator"}  # 不通过→重跑
)

# 5. 编译
agent = graph.compile()  # 编译后才能执行
```

### 两个条件路由

**route_by_intent**：
- search/summarize/refine → coordinator（走完整综述流程）
- qa → analysis（跳过检索，直接用已有论文库回答）

**route_by_critic**：
- 通过 → novelty（继续下一步）
- 未通过 + 重试次数 < 2 → coordinator（反馈 + 重跑）
- 未通过 + 重试次数 ≥ 2 → novelty（强制通过，避免死循环）

## 五、ResearchState 的设计

`agent/state.py` — TypedDict，所有节点共享。

**为什么用 TypedDict 而不是 Pydantic？**
- LangGraph 要求 state 是 dict-like 的
- `total=False` 表示所有字段都是可选的（节点只返回自己修改的字段）
- Pydantic 的 PaperMeta 只用于单篇论文的结构化

**数据流向**：
- query 在 intent 产生，一路传递到最后
- papers 在 search 产生，analysis/novelty/decomposition/trend/profile 都消费
- summary 在 writing 产生，critic 消费评估，novelty 消费分析
- critic_feedback 在 critic 产生，写回 coordinator 重新规划

## 六、面试高频问题

### Q1: 为什么用 LangGraph 而不是自己写 if-else 编排？

LangGraph 提供：
- **状态自动管理**：节点返回的 dict 自动 merge 回全局 state
- **条件路由**：`add_conditional_edges` 声明式定义分支逻辑
- **可观测性**：每个节点的输入/输出都可以追踪（LangSmith）
- **持久化**：支持 checkpoint，中断后可恢复

自己写 if-else 的话，状态传递、错误处理、可观测性都要自己搞，容易出错。

### Q2: Critic 反馈给 Coordinator 重跑，会不会死循环？

不会，MAX_RERUN=2 限制了最多重跑 2 次，超过就强制通过。

### Q3: 为什么 QA 模式跳过检索直接走 Analysis？

因为 QA 是基于**已有论文库**回答问题，不需要新检索。用户问"CIL 中 replay 方法的主要挑战是什么"，
系统从已索引的论文中用向量检索找相关 chunk，然后 LLM 生成回答。

### Q4: Trend Agent 为什么不复用综述检索的论文？

综述检索只返回 10 篇，且经过引用数筛选，年份只看 2022+。
这 10 篇的年份分布不代表全领域趋势。Trend 需要领域级统计数据
（S2 的 total 字段），所以必须单独做一轮大规模检索。

### Q5: State 为什么用 total=False？

因为不是每个节点都会产生所有字段。intent_node 只返回 intent，
coordinator_node 只返回 search_queries 和 focus_areas。
total=False 让每个节点只返回自己负责的字段，LangGraph 自动 merge。
